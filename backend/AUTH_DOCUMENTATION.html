1. System Overview & Architecture

The Dormocracy authentication system implements a stateless JWT-based authentication mechanism with refresh-token rotation. It follows NestJS best practices using a modular architecture with clear separation of concerns.

High-Level Architecture

┌─────────────┐ HTTP ┌──────────────────┐ Service ┌──────────────────┐ │ │ ──────────▶ │ AuthController │ ───────────▶ │ AuthService │ │ Client │ │ (4 endpoints) │ │ (business logic)│ │ (Browser) │ ◀────────── │ │ ◀─────────── │ │ └─────────────┘ JSON + └──────────────────┘ └────────┬─────────┘ Cookie │ │ Prisma ORM ▼ ┌──────────────────┐ ┌──────────────────┐ │ JwtStrategy │ │ PostgreSQL │ │ (Passport.js) │ │ Database │ │ validates token │ │ (User table) │ └──────────────────┘ └──────────────────┘ │ ▼ ┌──────────────────┐ │ JwtAuthGuard │ ← protects routes │ RolesGuard │ ← RBAC enforcement └──────────────────┘

Key Design Decisions

  • Dual-token system: Short-lived access tokens (JWT) + long-lived refresh tokens
  • Refresh token rotation: Every refresh generates a new pair of tokens, invalidating the old refresh token
  • httpOnly cookies: Refresh tokens are stored in secure, httpOnly cookies (not accessible via JavaScript)
  • Argon2 hashing: Both passwords and refresh tokens are hashed using Argon2 (winner of the Password Hashing Competition)
  • Role-Based Access Control (RBAC): Five roles — CITIZEN, MAYOR, MINISTER, PM, ADMIN

3. Database Schema (Prisma)

The authentication system relies on two core schema elements: the Role enum and the User model.

3.1 Role Enum

prisma/schema.prisma
enum Role {
  CITIZEN      // Default role for all new users
  MAYOR        // Room-level leader
  MINISTER     // Department-level role
  PM           // Project Manager
  ADMIN        // Full system access
}
ℹ️ How roles work: Every new user is assigned CITIZEN by default. Roles are stored in the JWT payload and checked by the RolesGuard on protected routes.

4. Auth Module Setup

src/auth/auth.module.ts
import { Module } from '@nestjs/common';
import { JwtModule } from '@nestjs/jwt';
import { PassportModule } from '@nestjs/passport';

import { AuthService } from './auth.service';
import { AuthController } from './auth.controller';
import { JwtStrategy } from './strategies/jwt.strategy';

@Module({
  imports: [
    PassportModule,                // Enables Passport integration
    JwtModule.register({}),        // Empty config — secrets provided at sign/verify time
  ],
  controllers: [AuthController],   // Registers HTTP endpoints
  providers: [AuthService, JwtStrategy],  // DI providers
  exports: [AuthService],          // Allows other modules to use AuthService
})
export class AuthModule { }

Line-by-Line Explanation

  • PassportModule — Integrates Passport.js into NestJS's dependency injection system. Required for AuthGuard to work.
  • JwtModule.register({}) — Registers the JWT service. Passing {} means no default secret is configured — instead, secrets are provided explicitly when calling signAsync() and verifyAsync(). This allows using different secrets for access vs. refresh tokens.
  • JwtStrategy — Passport strategy for validating access tokens. Must be listed as a provider so NestJS can inject ConfigService into it.
  • exports: [AuthService] — Other modules (e.g., UsersModule) can import AuthModule to use AuthService.

5. DTOs — Data Transfer Objects

DTOs define the shape and validation rules for incoming request bodies. They use class-validator decorators, which are enforced by NestJS's global ValidationPipe.

import { IsEmail, IsString, MinLength } from 'class-validator';

export class RegisterDto {

6.2 Token Signing — signTokens()

private async signTokens(userId: string, email: string, role: Role): Promise<Tokens> {
  // 1. Read secrets from environment
  const accessSecret  = this.config.getOrThrow<string>('JWT_ACCESS_SECRET');
  const refreshSecret = this.config.getOrThrow<string>('JWT_REFRESH_SECRET');

  // 2. Read TTLs (with defaults: 15 min access, 30 day refresh)
  const accessTtl  = Number(this.config.get<string>('JWT_ACCESS_TTL_SECONDS')  ?? 900);
  const refreshTtl = Number(this.config.get<string>('JWT_REFRESH_TTL_SECONDS') ?? 2592000);

  // 3. Build payload — this data is encoded into the JWT
  const payload = { sub: userId, email, role };

  // 4. Sign both tokens concurrently using different secrets
  const [accessToken, refreshToken] = await Promise.all([
    this.jwt.signAsync(payload, { secret: accessSecret,  expiresIn: accessTtl }),
    this.jwt.signAsync(payload, { secret: refreshSecret, expiresIn: refreshTtl }),
  ]);

  return { accessToken, refreshToken };
}
⚠️ Why two different secrets? Using separate secrets for access and refresh tokens ensures that an access token cannot be used as a refresh token (and vice versa). Even if an access token is intercepted, it cannot be used to generate new tokens.

6.3 Storing Refresh Token Hash

private async setRefreshTokenHash(userId: string, refreshToken: string) {
  const hash = await argon2.hash(refreshToken);   // Hash the raw JWT string
  await this.prisma.user.update({
    where: { id: userId },
    data:  { refreshTokenHash: hash },            // Store hash in DB
  });
}
ℹ️ Why hash the refresh token? If the database is compromised, attackers get only hashes — not usable tokens. Argon2 is deliberately slow, making brute-force attacks impractical.

6.4 Registration — register()

async register(dto: RegisterDto) {
  // 1. Validate department exists
  const department = await this.prisma.department.findUnique({
    where: { name: dto.departmentName },
  });
  if (!department) {
    throw new BadRequestException(`Department '${dto.departmentName}' does not exist`);
  }

  // 2. Hash the password with Argon2
  const passwordHash = await argon2.hash(dto.password);

  // 3. Run all creates in a transaction (atomic — all succeed or all fail)
  const user = await this.prisma.$transaction(async (tx) => {

    // 3a. Find or create the room
    const room = await tx.room.upsert({
      where: {
        departmentId_roomNumber: {
          departmentId: department.id,
          roomNumber: dto.roomNumber,
        },
      },
      update: {},       // Room exists → no changes
      create: {         // Room doesn't exist → create it
        departmentId: department.id,
        roomNumber: dto.roomNumber,
      },
    });

    // 3b. Ensure a ChatRoom exists for this room
    const chatRoom = await tx.chatRoom.upsert({
      where:  { roomId: room.id },
      update: {},
      create: { type: 'ROOM', roomId: room.id },
    });

    // 3c. Create the user record
    const createdUser = await tx.user.create({
      data: {
        username: dto.username,
        email:    dto.email,
        password: passwordHash,     // Store hashed password
        role:     Role.CITIZEN,     // Default role
        roomId:   room.id,
      },
    });

    // 3d. Add user to the chat room
    await tx.chatRoomMember.create({
      data: { chatRoomId: chatRoom.id, userId: createdUser.id },
    });

    return createdUser;
  });

  // 4. Generate tokens and store refresh hash
  const tokens = await this.signTokens(user.id, user.email, user.role);
  await this.setRefreshTokenHash(user.id, tokens.refreshToken);

  return tokens;  // { accessToken, refreshToken }
}
✅ Transaction safety: The $transaction() block ensures that if any step fails (e.g., duplicate email, database error), all changes are rolled back automatically. The user, room, chat room, and membership are either all created or none are.

6.5 Login — login()

async login(dto: LoginDto) {
  // 1. Find user by email
  const user = await this.prisma.user.findUnique({ where: { email: dto.email } });
  if (!user) throw new UnauthorizedException('Invalid credentials');

  // 2. Verify password against stored Argon2 hash
  const ok = await argon2.verify(user.password, dto.password);
  if (!ok) throw new UnauthorizedException('Invalid credentials');

  // 3. Generate new token pair
  const tokens = await this.signTokens(user.id, user.email, user.role);
  await this.setRefreshTokenHash(user.id, tokens.refreshToken);

  return tokens;
}
ℹ️ Security note: The same generic error message "Invalid credentials" is returned for both "user not found" and "wrong password" scenarios. This prevents attackers from enumerating valid email addresses.

6.6 Token Refresh — refresh()

async refresh(refreshToken: string) {
  const refreshSecret = this.config.getOrThrow<string>('JWT_REFRESH_SECRET');

  // Step 1: Verify the JWT signature and expiration
  let payload: { sub: string; email: string; role: Role };
  try {
    payload = await this.jwt.verifyAsync(refreshToken, {
      secret: refreshSecret,
    });
  } catch {
    throw new UnauthorizedException('Invalid refresh token');
  }

  const userId = payload.sub;

  // Step 2: Load user and check if refresh token hash exists
  const user = await this.prisma.user.findUnique({ where: { id: userId } });
  if (!user || !user.refreshTokenHash) {
    throw new UnauthorizedException('Access denied');
  }

  // Step 3: Verify the token matches the stored hash
  const ok = await argon2.verify(user.refreshTokenHash, refreshToken);
  if (!ok) {
    throw new UnauthorizedException('Access denied');
  }

  // Step 4: Rotate — generate new tokens and update the hash
  const tokens = await this.signTokens(user.id, user.email, user.role);
  await this.setRefreshTokenHash(user.id, tokens.refreshToken);

  return tokens;
}
🔒 Refresh Token Rotation — Three Layers of Security:
  1. JWT verification: Checks the cryptographic signature and expiry date
  2. Existence check: If refreshTokenHash is null, the user has logged out
  3. Hash comparison: The token must match the hash in the database — prevents replay of old tokens

6.7 Logout — logout()

async logout(userId: string) {
  await this.prisma.user.update({
    where: { id: userId },
    data:  { refreshTokenHash: null },   // Nullify → no refresh possible
  });
  return { success: true };
}

Setting refreshTokenHash to null effectively invalidates all refresh tokens for this user. The access token will continue to work until it expires (typically 15 minutes), but no new tokens can be generated.

7.1 Cookie Helper Methods

// Sets the refresh token as a secure, httpOnly cookie
private setRefreshCookie(reply: FastifyReply, refreshToken: string) {
  reply.setCookie('refresh_token', refreshToken, {
    httpOnly: true,       // Not accessible via JavaScript (XSS protection)
    secure: true,         // Only sent over HTTPS
    sameSite: 'none',     // Required for cross-site cookies (frontend ≠ backend domain)
    path: '/',            // Cookie available on all routes
    maxAge: 60 * 60 * 24 * 30,  // 30 days (in seconds)
  });
}

// Clears the cookie on logout
private clearRefreshCookie(reply: FastifyReply) {
  reply.clearCookie('refresh_token', { path: '/' });
}

7.4 Refresh Endpoint

@Post('refresh')  // POST /auth/refresh
async refresh(
  @Req() req: FastifyRequest,                         // Access to request (cookies)
  @Res({ passthrough: true }) reply: FastifyReply
) {
  const refreshToken = (req.cookies as any)?.refresh_token;  // Read from cookie
  if (!refreshToken) throw new UnauthorizedException('Missing refresh token');

  const tokens = await this.auth.refresh(refreshToken);
  this.setRefreshCookie(reply, tokens.refreshToken);   // Set new cookie (rotation)
  return { accessToken: tokens.accessToken };
}

7.5 Logout Endpoint (Protected)

@UseGuards(JwtAuthGuard)  // ← Requires valid access token
@Post('logout')            // POST /auth/logout
async logout(
  @Req() req: any,
  @Res({ passthrough: true }) reply: FastifyReply
) {
  await this.auth.logout(req.user.sub);          // req.user comes from JwtStrategy
  this.clearRefreshCookie(reply);                 // Delete the cookie
  return { success: true };
}
ℹ️ @Res({ passthrough: true }) — The passthrough option tells NestJS to still handle the response (serialization, status codes, etc.) even though we're injecting reply. Without it, NestJS would expect you to manually call reply.send().

8. JWT Strategy (Passport.js)

The JWT strategy tells Passport how to extract and validate access tokens from incoming requests.

src/auth/strategies/jwt.strategy.ts
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { ExtractJwt, Strategy } from 'passport-jwt';
import { ConfigService } from '@nestjs/config';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor(config: ConfigService) {
    super({
      // Extract token from: "Authorization: Bearer <token>"
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),

      // Use the ACCESS secret (not refresh secret)
      secretOrKey: config.getOrThrow<string>('JWT_ACCESS_SECRET'),
    });
  }

  // Called AFTER token is verified. Return value → req.user
  validate(payload: any) {
    return payload;  // { sub, email, role, iat, exp }
  }
}

How It Works

  1. When a request hits a route protected by JwtAuthGuard, Passport intercepts it
  2. ExtractJwt.fromAuthHeaderAsBearerToken() extracts the JWT from the Authorization header
  3. Passport verifies the token's signature using JWT_ACCESS_SECRET and checks expiration
  4. If valid, the validate() method is called with the decoded payload
  5. The return value of validate() is attached to req.user, making it available in controllers and guards

9.1 JwtAuthGuard

9.2 RolesGuard — RBAC

src/auth/guards/roles.guard.ts
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { ROLES_KEY } from '../decorators/roles.decorator';
import { Role } from '@prisma/client';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) { }

  canActivate(ctx: ExecutionContext): boolean {
    // 1. Read the @Roles() metadata from the handler or class
    const required = this.reflector.getAllAndOverride<Role[]>(ROLES_KEY, [
      ctx.getHandler(),    // Check method-level decorator first
      ctx.getClass(),      // Then class-level decorator
    ]);

    // 2. If no roles are specified, allow access
    if (!required || required.length === 0) return true;

    // 3. Get the user from the request (set by JwtStrategy)
    const req = ctx.switchToHttp().getRequest();
    const user = req.user;

    // 4. Check if user's role is in the required roles
    return required.includes(user.role);
  }
}

How to Use Guards Together

// Example: Only ADMIN and PM can access this endpoint
@UseGuards(JwtAuthGuard, RolesGuard)   // First authenticate, then authorize
@Roles(Role.ADMIN, Role.PM)             // Specify allowed roles
@Post('sensitive-action')
async doSomething() {
  // Only reachable by authenticated ADMIN or PM users
}
⚠️ Guard order matters: JwtAuthGuard must come before RolesGuard in the @UseGuards() array. The RolesGuard depends on req.user, which is only set after JWT validation.

async function bootstrap() {
